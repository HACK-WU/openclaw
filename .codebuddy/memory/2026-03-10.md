## 群聊测试修复完成

修复了 `ui/src/ui/controllers/group-chat.test.ts` 中 8 个失败的测试（全部在 initiator summary mechanism 下）。

### 根因

1. **`vi.runAllTimers()` 无限循环** — `scheduleSummaryCheck` 在 `isConversationBusy` 时递归设置 1000ms polling timer，导致 `vi.runAllTimers()` 跑满 10000 次上限
2. **异步 `detectAndForwardMentions` 未等待** — 源码改为使用 `resolveGroupMeta()` 异步获取元数据，但测试中 `handleGroupMessageEvent` 的 fire-and-forget 异步调用未被等待
3. **Mock client 不支持 `group.info`** — `resolveGroupMeta` 走 `fetchGroupMeta` 时调用 `host.client.request("group.info", ...)` 但 mock 只返回 `{}`
4. **缺少 `activeGroupId`** — 部分测试未设置 `host.activeGroupId = "g1"`，导致 `appendSystemMessageToUI` 的守卫拦截

### 修复方案

- `makeMockClient()` 改为对 `"group.info"` 请求返回完整的 `GroupSessionMeta`
- 提取 `DEFAULT_TEST_META` 常量共享给 `makeHost()` 和 mock
- 添加 `flushAsync()` 辅助函数，使用 `vi.advanceTimersByTimeAsync()` 同时刷新 microtask 和 fake timer
- `afterEach` 从 `vi.runAllTimers()` 改为 `vi.clearAllTimers()`（不执行，只清理）
- 所有使用 `handleGroupMessageEvent` + 检查异步效果的测试改为 `async` + `await flushAsync()`
- 需要等待 summary 触发的测试改用 `await vi.advanceTimersByTimeAsync()`
- "cancels automatic summary timer" 测试：用 `mockClear()` 确保只检查 cancel 之后的调用

最终结果：45 passed / 0 failed，lint 0 diagnostics。

## 更新群聊汇总设计文档：观察式等待

更新了 `docs/design/group-chat-initiator-summary.md`，将 `executeSummaryFlow` 从"投递 → 盲等 5 秒 → 汇总"改为两阶段设计：

- **阶段一（投递）**：投递待处理消息后，如果有实际投递，重新进入 `scheduleSummaryCheck` 等待循环
- **阶段二（汇总）**：等所有被投递的 agent 回复完毕（或超时），UI 再次空闲后，才发送汇总给 initiator

核心改动：

- 移除 `PRE_SUMMARY_DELIVER_MS`（5 秒盲等）
- `deliverPendingMentions` 返回 `boolean`，投递后清空已投递的非 initiator 消息
- `executeSummaryFlow` 根据 `hasDelivered` 决定是重新等待还是直接汇总
- 新增 4.1~4.4 小节详细说明设计理由、对比分析和防无限循环保障
- 更新完整流程示例，展示两阶段设计的效果

## Bridge Agent 方案可行性分析

完成了群聊集成外部 CLI 编码工具（Claude Code、OpenCode 等）的 Bridge Agent 方案可行性分析，文档保存在 `docs/design/group-chat-bridge-agent.md`。

### 核心结论

- **高度可行**：现有架构分层设计使替换推理层极其简单
- **最小侵入**：核心改动仅 ~10 行（`agent-trigger.ts` 分叉判断），新增 ~760 行独立模块
- **完全复用**：消息分发、transcript、广播、防循环、前端 @mention 转发、发起者汇总 — 零改动复用
- **向后兼容**：通过 `GroupMember.bridge?` 可选字段扩展

### 关键设计

- 在 `triggerAgentReasoning()` 中判断 `member.bridge` 是否存在来分叉
- Bridge Agent 走 `triggerBridgeAgent()` → `BridgeProcessManager` → CLI 进程
- CLI 进程通过 stdin/stdout 管道通信，stdout 流式输出转为 `group.stream` delta 事件
- Bridge Agent 不受 read-only tool policy 限制
- 支持 Claude Code `--print` 模式（MVP）和交互式管道模式（增强）
