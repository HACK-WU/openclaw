# 群聊 @提及 处理流程详解

本文档详细描述群聊中 @提及 的完整处理流程，包括前端发送、后端调度、以及回复渲染的机制。

---

## 一、概念定义

### 1.1 角色

| 角色          | 说明                       |
| ------------- | -------------------------- |
| **Owner**     | 群聊创建者，即用户本人     |
| **Agent**     | 群聊成员中的 AI 智能体     |
| **Assistant** | 群聊中的主助手（特殊角色） |

### 1.2 提及类型

| 类型          | 示例                  | 说明                     |
| ------------- | --------------------- | ------------------------ |
| **@全体成员** | `@all` 或 `@全体成员` | 触发所有 Agent           |
| **@特定成员** | `@dev` `@test_2`      | 触发指定的 Agent         |
| **专用提及**  | 单独一行只有 @提及    | 用于路由，会从消息中移除 |
| **行内提及**  | 消息内容中包含 @提及  | 仅用于显示，不触发路由   |

### 1.3 消息模式

| 模式                  | 行为                          |
| --------------------- | ----------------------------- |
| **Unicast（单播）**   | 无 @提及 时，只触发 Assistant |
| **Broadcast（广播）** | 无 @提及 时，触发所有成员     |

---

## 二、Owner 发送消息的处理流程

### 2.1 前端：解析 @提及

当 Owner 在输入框中输入消息并发送时：

**步骤 1：文本解析**

前端使用正则表达式扫描消息文本，提取所有 @xxx 格式的提及：

- 如果识别到 `@all` 或 `@全体成员`，标记 `hasAllMention = true`
- 其他 @提及 会匹配群成员列表，验证是否为有效成员

**步骤 2：展开 @全体成员**

如果存在 `@all` 标记：

- 遍历群成员列表
- 将所有成员的 `agentId` 加入 `mentions` 数组
- 最终 `mentions` 包含所有成员 ID，而非字符串 "all"

**步骤 3：去重**

使用 Set 数据结构对 `mentions` 数组去重，确保每个成员只出现一次。

**步骤 4：发送请求**

将消息内容和 mentions 数组通过 WebSocket 发送给后端：

```
请求结构：
{
  groupId: "群组ID",
  message: "消息文本",
  mentions: ["dev", "test_2", "test_3"]  // 已展开的成员列表
}
```

### 2.2 前端：预测待响应 Agent

在发送请求的同时，前端会预测哪些 Agent 将被触发：

**预测逻辑：**

1. 有 mentions 数组 → 返回 mentions 中的成员
2. 无 mentions + 单播模式 → 返回 Assistant
3. 无 mentions + 广播模式 → 返回所有成员

这些预测的 Agent 会被加入 `groupPendingAgents` 集合，用于在 UI 上显示"等待响应"的指示器。

### 2.3 后端：接收并调度

后端收到 `group.send` 请求后的处理流程：

**步骤 1：验证**

- 验证 groupId 是否有效
- 验证群组是否已归档
- 验证 mentions 中的 agentId 是否都是群成员

**步骤 2：重置链状态**

Owner 发送新消息时，后端会重置"对话链状态"：

- 清除之前的 Agent 间转发计数
- 开始新一轮的对话

**步骤 3：解析调度目标**

后端调用调度解析函数，确定哪些 Agent 需要被触发：

| 条件                 | 调度模式       | 目标                            |
| -------------------- | -------------- | ------------------------------- |
| mentions 不为空      | mention 模式   | mentions 中的成员（排除发送者） |
| mentions 为空 + 单播 | unicast 模式   | 仅 Assistant                    |
| mentions 为空 + 广播 | broadcast 模式 | 所有成员（排除发送者）          |

### 2.4 后端：并行 vs 串行触发

这是关键的调度差异：

**并行触发（Broadcast 模式）：**

当调度模式为 `broadcast` 时：

- 所有目标 Agent **同时**开始推理
- 使用 `Promise.allSettled` 并行执行
- 每个 Agent 独立生成回复，互不等待

**串行触发（Unicast/Mention 模式）：**

当调度模式为 `unicast` 或 `mention` 时：

- 目标 Agent **依次**触发
- 等待前一个 Agent 完成回复后，再触发下一个
- 每次触发都会更新对话链状态

**为什么区分并行和串行？**

- Broadcast（广播）模拟真实群聊：所有人同时看到消息
- Mention（提及）是有针对性的提问：让被提及者按顺序发言
- Unicast（单播）是一对一对话：只需 Assistant 回复

## 三、Agent 自动 @其他 Agent 的处理流程

### 3.1 Agent 回复中的 @提及

当 Agent 完成推理并生成回复后：

**后端处理：**

1. 将回复内容写入消息记录（transcript）
2. 通过 WebSocket 广播 `group.message` 事件给前端
3. 广播 `group.stream` 事件（状态为 `final`）

**注意：** 后端**不会**自动解析 Agent 回复中的 @提及 来触发其他 Agent。

### 3.2 前端：检测专用提及并转发

这是 Agent 间自动对话的核心机制。

**触发时机：**

前端收到 `group.message` 事件后，会调用 `detectAndForwardMentions` 函数。

**检测逻辑：**

1. 只处理 Agent 发送的消息（跳过 Owner 和 System）
2. 按行解析消息内容
3. 识别"专用提及行"——整行只有 @提及，没有其他内容

**专用提及示例：**

```
这是一条普通消息
@dev @test_2    ← 这一行是专用提及（只有 @mentions）
这是另一条普通消息
```

在上述示例中，只有第二行的 `@dev @test_2` 会被识别为路由指令。

**非专用提及示例：**

```
请 @dev 帮忙看看    ← 这不是专用提及，因为有其他内容
@dev 请回答        ← 这不是专用提及，因为有其他内容
```

这些行内的 @提及 仅用于显示高亮，不会触发路由。

### 3.3 前端：转发逻辑

当检测到专用提及后：

**步骤 1：过滤无效提及**

- 只保留群成员列表中存在的 agentId
- 排除 Initiator（发起者，防止被意外触发）
- 排除消息发送者自己

**步骤 2：更新对话链状态**

- 增加转发计数
- 记录当前时间戳
- 将发送者加入 `initiators` 列表（去重）
- 将被提及者加入 `pendingAgents` 集合

**步骤 3：移除专用提及行**

专用提及行是路由指令，不是消息内容。转发时会移除这些行，只保留实际内容。

**步骤 4：发送转发请求**

调用后端 `group.send` 接口，携带：

- 原始消息内容（已移除专用提及行）
- 被提及的 agentId 列表
- 发送者设置为原 Agent（不是 Owner）
- `skipTranscript: true` 标记（跳过重复写入）

### 3.4 后端：处理 Agent 转发

后端收到 Agent 发起的转发请求后：

**特殊处理：**

1. 检查后端对话链限制（防止无限循环）
   - 最多 20 次转发
   - 最长 5 分钟持续时间
2. 如果超限，返回 429 错误
3. 正常情况下，按 mention 模式串行触发目标 Agent

## 四、回复的渲染机制

### 4.1 后端：流式广播

Agent 推理过程中，后端会持续发送流式事件：

**事件类型：**

| 事件            | 状态    | 说明                         |
| --------------- | ------- | ---------------------------- |
| `group.stream`  | delta   | 增量文本，包含当前生成的内容 |
| `group.stream`  | final   | 推理完成，包含完整消息对象   |
| `group.stream`  | error   | 推理出错                     |
| `group.stream`  | aborted | 被中止                       |
| `group.message` | -       | 最终消息，写入消息列表       |

**广播时机：**

- delta：每当 Agent 生成新文本片段时
- final：Agent 完成推理后
- message：消息写入记录后

### 4.2 前端：接收流式事件

**流缓冲机制：**

前端使用缓冲区 + 定时器来处理流式更新：

1. 收到 delta 事件时，将文本存入缓冲区
2. 启动 50ms 定时器
3. 定时器触发时，批量更新 UI 状态
4. 避免过于频繁的渲染刷新

**状态管理：**

- `groupStreams`：Map 类型，key 是 agentId，value 是当前流式文本
- `groupPendingAgents`：Set 类型，存储待响应的 agentId
- `groupMessages`：Array 类型，存储已完成的消息

### 4.3 前端：渲染策略与发送者识别

**前端如何识别消息发送者？**

前端通过后端发送的事件中的发送者信息来识别：

| 事件类型        | 发送者信息字段 | 结构                                                     |
| --------------- | -------------- | -------------------------------------------------------- |
| `group.message` | `sender`       | `{ type: "agent"\|"owner"\|"system", agentId?: string }` |
| `group.stream`  | `agentId`      | 字符串（直接是 agentId）                                 |

**关键逻辑：**

1. **已完成消息**：从 `group.message` 事件的 `sender` 字段读取
   - `sender.type === "agent"` → 显示 Agent 名称
   - `sender.type === "owner"` → 显示 "You"
   - `sender.agentId` → 用于查找 Agent 详细信息（emoji、名称等）

2. **流式消息**：从 `group.stream` 事件的 `agentId` 字段读取
   - 直接使用 `payload.agentId` 作为 key
   - 存储在 `groupStreams` Map 中：`Map<agentId, streamData>`

**渲染映射流程：**

```
后端广播 group.message
      ↓
前端接收事件
      ↓
提取 sender.agentId（如果是 agent 类型）
      ↓
查找 Agent 信息（emoji、名称）
      ↓
渲染消息气泡，显示正确的发送者
```

**发送者身份错位 Bug 的可能原因：**

如果后端发送的 `sender.agentId` 或 `payload.agentId` 是错误的：

- 前端会忠实地显示错误的发送者
- 因为前端完全依赖后端提供的发送者信息

**同时渲染 vs 一起渲染：**

答案是**同时渲染，独立气泡**。

每个 Agent 的回复都会：

1. 独立渲染为一个消息气泡
2. 按收到顺序追加到消息列表
3. 流式更新时，气泡内容实时变化

**并行回复的渲染顺序：**

当多个 Agent 同时被触发时：

- 回复完成的顺序取决于各自的推理速度
- 先完成的先显示，后完成的后显示
- 没有固定的顺序保证

**流式气泡：**

在 Agent 推理过程中：

- 显示一个"正在输入"的气泡
- 气泡内容随着 delta 事件实时更新
- 收到 final 事件后，气泡固定为最终内容

## 五、Agent @多个成员 vs Owner @多个成员

### 5.1 逻辑对比

| 维度             | Owner @多个成员                    | Agent @多个成员            |
| ---------------- | ---------------------------------- | -------------------------- |
| **触发方式**     | 直接通过后端调度                   | 通过前端检测专用提及       |
| **处理模式**     | 并行（broadcast）或串行（mention） | 始终串行（mention 模式）   |
| **链状态**       | 重置新一轮                         | 延续当前链                 |
| **专用提及处理** | 无（Owner 消息没有路由概念）       | 移除专用提及行，仅转发内容 |
| **发送者标记**   | Owner                              | 原 Agent                   |

### 5.2 关键差异

**Owner @多个成员：**

- mentions 数组直接发送给后端
- 后端根据模式决定并行或串行
- 所有被提及者同时看到消息

**Agent @多个成员：**

- 需要使用"专用提及"格式（单独一行）
- 前端检测后才会转发
- 串行触发，一个一个回复

### 5.3 示例

**Owner @dev @test_2：**

```
Owner: @dev @test_2 请分析这个问题
      ↓
后端收到 mentions: ["dev", "test_2"]
      ↓
串行触发：dev 先回复 → test_2 再回复
```

**Agent @dev @test_2（专用提及）：**

```
Agent_A: 我认为需要其他人帮忙
         @dev @test_2
      ↓
前端检测到专用提及
      ↓
转发请求：mentions: ["dev", "test_2"], sender: Agent_A
      ↓
串行触发：dev 先回复 → test_2 再回复
```

## 六、多个回复 @相同成员的处理

### 6.1 场景描述

```
Owner: @dev @test_2 请讨论这个问题
dev: 我认为... @test_3  ← 专用提及 test_3
test_2: 我也认为... @test_3  ← 也专用提及 test_3
```

### 6.2 处理机制

**pendingAgents 集合：**

前端使用 Set 数据结构跟踪待响应的 Agent：

- Set 自动去重，多次添加相同 agentId 只保留一个
- 当 Agent 回复时，从 Set 中移除

**处理流程：**

1. dev 回复，检测到 `@test_3`
   - 将 `test_3` 加入 `pendingAgents`
   - 发送转发请求，触发 test_3

2. test_2 回复，也检测到 `@test_3`
   - `test_3` 已在 `pendingAgents` 中（Set 自动去重）
   - 再次发送转发请求

3. test_3 收到两次触发请求
   - 后端会检查对话链限制
   - 如果第一次触发后 test_3 还没回复，第二次触发会被处理
   - 但如果 test_3 已经回复过，第二次触发会被防循环机制拒绝

### 6.3 防重复触发机制

**后端防循环：**

- `maxRounds`：每条 Owner 消息最多触发轮次
- `maxConsecutive`：同一 Agent 最多连续被触发次数

**前端 pendingAgents：**

- 记录已触发但尚未回复的 Agent
- 用于 UI 显示"等待响应"指示器
- 回复到达后自动移除

**实际效果：**

即使多个 Agent 都 @了同一个成员，该成员：

- 最多被触发 `maxConsecutive` 次（默认配置）
- 触发计数会在连续被触发时累加
- 如果中间有其他 Agent 回复，计数会重置

## 七、完整流程图

### 7.1 Owner @全体成员 流程

```
Owner 输入: @全体成员 明天开会
      ↓
前端解析: mentions = [dev, test_2, test_3]
      ↓
前端预测: pendingAgents = {dev, test_2, test_3}
      ↓
发送请求: group.send {mentions: [dev, test_2, test_3]}
      ↓
后端调度: broadcast 模式
      ↓
并行触发: dev || test_2 || test_3  (同时开始)
      ↓
流式广播: group.stream (delta) x 多个
      ↓
前端渲染: 多个流式气泡同时更新
      ↓
完成广播: group.stream (final) + group.message
      ↓
前端固定: 消息按完成顺序追加到列表
```

### 7.2 Agent 自动 @其他 Agent 流程

```
Agent_A 回复完成
      ↓
后端广播: group.message {sender: Agent_A, content: "..."}
      ↓
前端接收: handleGroupMessageEvent
      ↓
检测提及: detectAndForwardMentions
      ↓
解析内容: 识别专用提及行 "@Agent_B"
      ↓
过滤验证: Agent_B 是群成员，不是 Initiator
      ↓
更新状态: pendingAgents.add(Agent_B)
      ↓
发送转发: group.send {sender: Agent_A, mentions: [Agent_B]}
      ↓
后端调度: mention 模式，串行触发
      ↓
Agent_B 推理 → 回复 → 广播
      ↓
前端接收 → 渲染
```

## 八、边界情况

### 8.1 Agent @自己

- 前端检测时会排除发送者自己
- `@自己` 不会触发任何路由

### 8.2 Agent @Initiator

- Initiator（发起者）会从匹配池中排除
- 即使被 @，也不会触发路由
- 防止 Initiator 被反复拉回对话

### 8.3 Agent @非群成员

- 非群成员不在匹配列表中
- @非群成员 会被忽略，不触发路由

### 8.4 对话链超限

- 后端返回 429 错误
- 前端显示系统消息提示用户
- 需要 Owner 发送新消息才能继续

### 8.5 中止请求

- Owner 点击"停止"按钮
- 后端中止所有正在进行的 Agent 推理
- 已生成的内容会保留

## 九、性能考量

### 9.1 并行触发的开销

- 所有 Agent 同时推理，AI API 并发压力大
- 大群组（10+ Agent）可能导致资源瓶颈
- 建议：限制群组成员数量，或使用队列调度

### 9.2 流式更新的频率

- 前端使用 50ms 定时器批量更新
- 避免每次 delta 都触发渲染
- 多 Agent 并行流式时，UI 更新压力更大

### 9.3 消息列表增长

- 长时间对话会产生大量消息
- 前端需要虚拟滚动或分页加载
- 后端提供 `before` 参数支持历史加载

## 十、总结

| 场景               | 触发方式     | 调度模式 | 渲染方式       |
| ------------------ | ------------ | -------- | -------------- |
| Owner @全体成员    | 后端直接调度 | 并行     | 多气泡同时更新 |
| Owner @多个成员    | 后端直接调度 | 串行     | 按完成顺序     |
| Agent @多个成员    | 前端检测转发 | 串行     | 按完成顺序     |
| 多 Agent @同一成员 | 前端去重     | 串行     | 防循环限制     |

**核心原则：**

1. Owner 消息触发新一轮对话
2. Agent 间转发延续当前对话链
3. 并行触发适用于广播场景
4. 串行触发适用于点名场景
5. 防循环机制保护系统稳定
