# 群聊创建对话框设计

## 背景

当前创建群聊时，智能体列表需要先勾选复选框，才会显示类型下拉框。这种交互方式增加了用户操作步骤，不够直观。

## 设计目标

1. 简化用户操作流程，让用户可以直接看到并选择每个智能体的类型
2. 支持项目配置的路径验证
3. 完整的国际化支持

---

## 角色类型说明

群成员支持以下 3 种角色：

| 角色            | 说明                                             |
| --------------- | ------------------------------------------------ |
| `assistant`     | 助手 - 普通 AI 助手，会参与消息路由              |
| `member`        | 成员 - 普通成员角色                              |
| `cli-assistant` | CLI 助手 - CLI Agent 桥接，用于连接外部 CLI 工具 |

## 行为规则

### 1. 类型下拉始终可见

每行都显示类型下拉框，不依赖勾选状态。

### 2. 默认值逻辑

- 第一个勾选的智能体默认角色为 `assistant`
- 其他智能体默认角色为 `member`

### 3. 未勾选时的下拉状态

- 显示默认值但处于"待确认"状态
- 可交互，但只在勾选后才真正生效
- 视觉上有区分（降低透明度或显示为浅色）

### 4. 角色切换行为

- 已勾选行：切换角色立即更新已选列表
- 未勾选行：切换角色仅记录为"选择意向"，勾选后使用该值

### 5. 路径验证

Project Directory 和 Project Docs 字段需要实时验证：

#### 验证规则

| 字段              | 验证内容                     | 验证失败表现              |
| ----------------- | ---------------------------- | ------------------------- |
| Project Directory | 目录是否存在                 | 输入框红色边框 + 错误提示 |
| Project Docs      | 每个文件是否存在（逗号分隔） | 输入框红色边框 + 错误提示 |

#### 触发时机

- 用户按回车键时触发验证
- 输入框失去焦点（blur）时触发验证

#### 错误处理

- 验证失败时，在输入框下方显示红色错误提示文字
- 创建按钮变为禁用状态（无法点击）
- 所有错误修复后才能创建群聊

#### 空值处理

- 字段为空时不进行验证（因为是 optional）
- 一旦填写内容，就必须通过验证才能创建

## 样式建议

### 未勾选时的下拉框

- 透明度：0.6
- 鼠标悬停时：透明度 0.8
- 背景色：浅灰色

### 已勾选时的下拉框

- 透明度：1
- 最小宽度：120px
- 字体大小：12px

### 错误状态

- 输入框边框：红色
- 错误图标：输入框右侧显示 ❌ 图标
- 错误文字：红色，12px，位于输入框下方
